CH32X035G8U6 USB-SPIの異常系の仕様書を策定中だが、

### **2.3 物理層の接続検知とデバウンス (tCCDebounce) の厳格化**

USB Type-Cコネクタの物理的な挿抜時には、接点のメカニカルなバウンドによる過渡的なサージや瞬断が数十ミリ秒にわたって発生する。これを考慮せず直ちにVBUSを印加すると、突入電流によるシステム電圧の降下（ブラウンアウト）や、接続先デバイスの起動失敗を誘発する。USB Type-C仕様では、状態遷移の誤検知を防ぐため tCCDebounce が規定されている 。

#### **2.3.1 正常系のアタッチ検知とVBUS印加**
CCピンの分圧電圧 (vRd) を監視し、電圧が vOPEN (未接続状態) から有効な検出範囲に降下したことを判定する。Default USB Power (Rp=36kΩ / Rd=5.1kΩ) の場合、適法な vRd の範囲は **0.25V 〜 1.50V** である 。 電圧降下を検知した場合、直ちにVBUSをONにしてはならない。必ず非同期のソフトウェアタイマーを起動し、**100ms 〜 200ms** の間、継続して vRd が規定範囲内に収まっていること (tCCDebounce) を確認してから、PA0 (Load Switch EN) をHighにし、VBUS (5V) を印加するシーケンスを実行すること 。

#### **2.3.2 異常系検知 (物理切断・ノイズによる瞬断) とコールドソケット要件**
USB Type-C規格において、未接続時にVBUSに5Vを出力し続ける「VBUS Hot」状態は厳格な違反行為である 。デバイスの非接続時、PA0は常にLowを維持し、VBUSを完全に遮断（コールドソケット）しなければならない。 運用中にCC電圧が vOPEN (\> 1.65V) に戻った場合、即座にデバイスが切断されたと判定してVBUSを遮断 (PA0 \= Low) する。ただし、静電気 (ESD) やEMIノイズによる瞬間的な電圧変動を考慮し、切断検知側にも 10ms 〜 20ms のデバウンスフィルタを設けることで、不要なバスリセットの発生を防ぐ設計とする。

**3\. USBホストコントローラ (USBFS) におけるプロトコル層異常系と回復シーケンス**

CH32X035のUSBFSモジュールをHostモードで駆動し、接続されたデバイスの列挙（Enumeration）からHIDレポートの定期的取得（Polling）までを行う。USB通信プロトコルは、ケーブルの品質、インピーダンス不整合、外部ノイズ、およびデバイス側のファームウェア実装の不備により、様々なエラー状態に陥る。本ブリッジは、これらのエラーをメインMCUに波及させることなく、自律的に回復を試みるステートマシンを備える。

### **3.1 列挙 (Enumeration) フェーズにおける異常系**

列挙フェーズでは、デバイスの素性を確認するための GET\_DESCRIPTOR 発行、アドレスを割り当てる SET\_ADDRESS、およびエンドポイント構成を確立する SET\_CONFIGURATION が連続して実行される。この期間はデバイスの初期化過渡期であり、最もエラーが頻発する。

#### **3.1.1 応答タイムアウトとトランザクションエラー (3 Strikes Rule)**

* **異常条件:** コントロール転送 (EP0) でセットアップパケットを送信後、デバイスから ACK、NAK、STALL のいずれのハンドシェイクPIDも返却されず、トランザクションがタイムアウトする。または、CRCエラーが発生する。  
* **根本原因:** バス上のノイズによるパケット破損、またはデバイス側のUSBコントローラの初期化遅延。  
* **回復シーケンス:** USB 2.0標準規格におけるエラーリカバリ手法に基づき、トランザクションエラーが発生した場合は**最大3回**まで同一パケットの再送を試みる（3 Strikes Rule）。3回連続で失敗した場合は、現在の列挙プロセスを放棄し、CH32X035からUSBバスリセット（D+ および D- を最低10ms以上SE0レベルに引き下げる）を発行して、状態をリセットした上で列挙を最初からやり直す。

#### **3.1.2 STALL 応答 (Unsupported Request / Endpoint Halted)**

* **異常条件:** デバイスから STALL PID が返却される 。

* **根本原因:** ホストが送信したリクエストをデバイスがサポートしていない（例: 存在しないString Descriptorの要求）、あるいはエンドポイントが論理的なエラーによりHalt状態に陥っている。  
* **回復シーケンス:**  
  1. **非必須リクエストの場合:** 製造元文字列（String Descriptor）などの取得時にSTALLが返った場合は、これを「正常な拒否」として許容し、パースをスキップして次の列挙ステップへ進む。  
  2. **必須リクエストの場合:** SET\_CONFIGURATION や GET\_DEVICE\_DESCRIPTOR など、列挙に不可欠なリクエストでSTALLが返った場合は致命的エラーである 。この場合、100msのリカバリウェイトを挟んだ後、USBバスリセットを発行してデバイス全体の初期化を再試行する。

#### **3.1.3 SET\_ADDRESS 後のリカバリタイム違反によるフリーズ**

* **異常条件:** SET\_ADDRESS コマンドの発行直後に実行される後続のリクエスト（デバイス記述子の再取得など）で、必ずタイムアウトやSTALLが発生する。  
* **根本原因:** USB 2.0規格 (セクション9.2.6.3) において、SET\_ADDRESS のステータスステージ（ACKの送受信）が完了した後、デバイスが内部で新しいアドレスを適用し、通信準備を整えるための\*\*「SetAddress() Recovery Interval」として2msが規定\*\*されている 。

* **回復シーケンス:** ファームウェアの列挙ステートマシンにおいて、SET\_ADDRESS コマンドの完了割り込みが発生した後、直ちに次のパケットを送信してはならない。必ずハードウェアタイマーを用いて **2ms (安全マージンを取り 5ms を推奨)** のウェイトタイムをブロッキングまたは非同期で設けること。特定の市販ゲームパッド（互換コントローラー等）は、この遅延がないと後続の要求を全損させる仕様的脆弱性を持つため、極めて重要である 。

#### **3.1.4 非標準・独自規格HIDデバイスの検知 (bInterfaceClass \= 0xFF)**

* **異常条件:** 列挙中に取得したコンフィギュレーション記述子を解析した結果、bInterfaceClass が標準の 0x03 (HID) ではなく、0xFF (Vendor Specific) となっている 。

* **根本原因:** Nintendo Switch用プロコントローラーや、PC用の特殊なゲーミングデバイスは、接続直後は独自のプロトコルモードで立ち上がる。  
* **回復シーケンス:** 本ブリッジは基本的に標準HIDのReport Descriptorをパースして動作するが、利便性向上のため、既知の Vendor Specific デバイス (VID/PIDで判別、例えばNintendo Switch互換機) に対しては、プロトコルを移行させるための特定の初期化マジックパケット（例: ボーレート変更コマンドやハンドシェイクパケット）をEP0または特定のエンドポイントへ送信し、標準のHIDモードへフォールバックさせる処理を実装する 。未知の Vendor Specific デバイスが接続された場合は、解析不能であるため列挙を意図的にアボートし、PA1およびPA2のLEDを交互点滅させて「非対応デバイス」エラーを視覚的に通知し、無駄なポーリングを停止する。

### **3.2 動作 (Polling) フェーズにおける異常系**

列挙が正常に完了し、Interrupt IN エンドポイントに対して GET\_REPORT に相当するINトークンを定期的に発行してデータを取得している最中のエラーハンドリング定義。

#### **3.2.1 NAKの連続応答と致命的ポーリングタイムアウト**

* **異常条件:** Interrupt IN エンドポイントにデータ要求（INトークン）を発行しても、デバイスから NAK (Not Acknowledge) が返り続ける。  
* **正常系との切り分け:** ユーザーがボタンやスティックを一切操作していない場合、USB HIDデバイスはUSBバスの帯域を節約するため、データパケットを送信せずに NAK を返すか、ゼロ長パケット (ZLP) を返す仕様が一般的である 。したがって、NAK の受信自体は通信エラーではない。

* **回復シーケンス:** デバイスの物理的なフリーズやプロトコルスタックのクラッシュを検知するため、**「1000ms (1秒) 連続して1度も有効なデータパケット (DATA0/DATA1) が受信できなかった場合」** を致命的なタイムアウトエラーと定義する 。 このタイムアウト状態に陥った場合は、直ちにUSBバスリセットを発行し、再列挙を試みる。3回リセットを試みてもタイムアウトが解消しない場合は、システムをディープリカバリ状態へ移行させ、PA0をLowにしてVBUSを物理的に切断し、2秒間待機した後に再度VBUSを印加する「ハードウェア再起動 (VBUS Toggle)」を実行する。

#### **3.2.2 バブルエラー (Babble Error)**

* **異常条件:** デバイスが、エンドポイント記述子で定義された最大パケットサイズ (wMaxPacketSize) を超える大量のデータを連続して送信し続け、フレームの終端 (EOF) を越えてバスを占有するバブル状態が発生する 。

* **根本原因:** USBケーブルの著しい劣化による信号反射、外部からの強力な電磁干渉 (EMI)、あるいはデバイス側の送信バッファ管理ファームウェアのバグ（暴走）によるプロトコル違反。バブルエラーはUSBバス全体を電気的にロックアップさせる致命的な障害である 。

* **回復シーケンス:** CH32X035のUSBFSステータスレジスタでバブルエラーフラグ (USBD\_STATUS\_BABBLE\_DETECTED 等に相当) を検知した場合、ハードウェアは該当のパケットを強制的に破棄する。ソフトウェア側では直ちに該当エンドポイントに対する処理を停止し、EP0コントロール転送を用いて CLEAR\_FEATURE (ENDPOINT\_HALT) をデバイスに送信し、Halt状態の回復を試みる 。この回復リクエストに対しても応答がない、または直後に再度バブルエラーが発生する場合は、即座にUSBバスリセットを発行する。

#### **3.2.3 トグルビットエラー (Data Toggle Desynchronization)**

* **異常条件:** ホスト側が期待するデータトグル (DATA0 / DATA1) と、デバイス側が送信してきたトグルビットが一致しない 。

* **根本原因:** 一時的なノイズによるパケットロスが発生し、ホスト側がパケットを受信できなかったにもかかわらず、デバイス側はACKを受け取ったと誤認してトグルビットを進めてしまった場合に発生する同期ズレ。  
* **回復シーケンス:** トグルエラーが発生した場合、ホストコントローラ（CH32X035）は受け取ったデータを破棄する。ソフトウェアはリトライカウンタをインクリメントし、再度INトークンを発行する。連続してトグルエラーが発生する場合は同期が完全に失われているため、コントロール転送で CLEAR\_FEATURE (ENDPOINT\_HALT) を送信し、ホスト側とデバイス側の両方のデータトグルシーケンスを DATA0 に強制リセットして同期を再確立する 。

#### **3.2.4 ゼロ長パケット (ZLP: Zero-Length Packet) の解釈**

* **異常条件:** IN要求に対して、デバイスからペイロードが0バイトのデータパケットが返却される。  
* **正常系への包含:** 一部のレガシーなジョイスティック（例: Competition Pro USB Joystick等）は、状態に変化がないことを通知するためにNAKではなくZLPを送信する実装となっている 。

* **回復シーケンス:** 転送完了割り込みにおいて、受信バイト長が0であった場合は通信エラーと誤認してはならない。このパケットは正常な「状態維持」レスポンスとして処理し、メインMCUに提供するための内部のゲームパッド状態バッファ（前回値）をそのまま保持し、何らエラー処理を行わずに次のポーリングサイクルへ移行する。

**4\. SPIスレーブ通信プロトコルと物理的同期の脆弱性対策**

CH32X035は、取得・解析したゲームパッドの入力状態を、メインMCU (RP2350等) へ高速に転送するためのデータパイプとしてSPI通信を提供する。通信は SPI Mode 0 ($CPOL=0, CPHA=0$) に準拠し、CH32X035は\*\*スレーブ (Slave)\*\*として動作する。

SPIプロトコルは、I2CやUSBと異なり、ハードウェアレベルでのハンドシェイク機構（ACK/NAKの返却）を持たず、エラー検知用のパリティやCRCもプロトコル自体には内包されていない 。そのため、電気的なノイズやタイミングのズレがシステム全体に壊滅的なデータの不整合をもたらす。本章では、この脆弱性を克服するための堅牢な同期メカニズムと異常系対策を定義する。

### **4.1 SPI同期における最大の脆弱性：「ビットシフト問題」**

SPIはマスターが生成するクロック (SCK) の立ち上がりエッジをカウントしてデータをシフトする。万が一、基板上の配線のクロストークや外部からの電気的ノイズ (EMI) によって SCK ラインに微小なヒゲ（グリッチ）が乗り、本来のクロックよりも多くカウントされてしまう、あるいは逆にスルーレートの不足でクロックが欠落すると、スレーブ内のシフトレジスタおよびビットカウンタが本来のバイト境界から逸脱する 。

**影響の深刻度:** 一度この「ビットシフト状態」に陥ると、後続のすべての転送においてデータが1ビットまたは数ビットずれた状態で解釈・送出されることになる。例えば、マスターが 0x0A (00001010) を送信しても、スレーブ側で1ビットずれて 0x14 (00010100) と誤認識される。USBのようなフレーム単位の自動リセット機構がないため、対策を講じなければ電源を切るまで永遠に正しいデータ交換ができなくなる致命的な異常である 。

### **4.2 CS (NSS) エッジ駆動による強制再同期メカニズム**

上記のビットシフト問題から自律的かつ即座に復帰するため、本ファームウェアでは **CS (Chip Select / NSS) ピンのトランジション（立ち上がり・立ち下がりエッジ）を絶対的な同期トリガーとするハードウェアおよびソフトウェアリセット機構** を必ず実装する 。

#### **4.2.1 正常系のトランザクション定義**

1. **Assertion:** Master (RP2350) が CSピンを High から Low に引き下げる。  
2. **Transfer:** MasterがSCKを生成し、MOSI経由でコマンドを送信、MISO経由でデータを受信する。  
3. **De-assertion:** 規定バイト数の送受信が完了後、Masterが CSピンを Low から High に戻す。

#### **4.2.2 異常系のリカバリ定義 (EXTI割り込みの併用)**

CH32X035の PA4 (CS) ピンをSPIモジュールのハードウェアNSSとして機能させつつ、**同時に EXTI (外部割り込み) の「立ち上がりエッジトリガー (Rising Edge)」および「立ち下がりエッジトリガー (Falling Edge)」** にマッピングする 。

* **CS 立ち下がり (トランザクション開始):**  
  内部状態をクリアし、通信準備を整える。  
* **CS 立ち上がり (トランザクション終了 / リカバリ):**  
  CSがHighに戻った瞬間、それが正常終了であっても、ノイズによる中途半端な中断であっても、以下のリカバリ処理を必ず実行し、ステートを強制リセットする。  
  1. **SPIモジュールの初期化:** 内部のSPI送受信シフトレジスタをフラッシュし、残留データを破棄する。  
  2. **ビットカウンタのクリア:** 内部のビットカウンタを強制的に 0 に戻す 。

  3. **DMAポインタの初期化:** データ転送にDMA (Direct Memory Access) のCircularモード等を利用している場合、DMAチャネルを一旦無効化し、転送カウンタ (NDTR) とメモリアドレスポインタをコマンド待機状態の初期位置へ巻き戻す 。これにより、次のCS立ち下がり時には必ず1バイト目の先頭ビットから正確に送受信が再開されることが保証される。

#### **4.2.3 ショートパケット（不完全フレーム）の破棄**

CSがHighになった（De-assertion）時点で、規定のバイト数（コマンド受信時は1バイト、データ送信時は3バイト等）のクロックを完全に受信しきっていなかった場合、そのトランザクションはノイズによる異常終了、またはマスター側の割り込み遅延によるアボートとみなす。この際、受信中のコマンドやデータは一切評価せず、完全に破棄してシステム状態への反映を防ぐ 。

### **4.3 コマンドとデータフェーズの分離によるプロトコル堅牢化**

汎用性と確実性を重視し、SPI通信は標準的な「コマンド送信フェーズ」と「データ受信フェーズ」の二段階分離方式を採用する。

1. **Phase 1 (Command):** Masterから1バイトのコマンドID (0x01 等) をMOSIへ送出する。スレーブは1バイト受信完了割り込みにてコマンドを解析し、次にMISOから送り出すべきデータを送信バッファにセットする。  
2. **Phase 2 (Data):** Masterはスレーブの処理時間を考慮した適切なウェイト（またはダミークロック数クロック分）の後、必要なデータ長分のダミークロックを送ってスレーブからデータを読み出す。  
3. 1つのコマンド・データ交換が完了するまで、MasterはCSをLowに保持し続ける。

#### **4.3.1 未定義・無効コマンドのハンドリング**

* **異常条件:** マスターのソフトウェアバグ、またはMOSIライン上のノイズにより、スレーブが定義表に存在しない不正なコマンドID (例: 0xFF, 0x00, 0x55 など) を受信した。  
* **回復シーケンス:** スレーブはコマンドIDを switch-case またはルックアップテーブルで厳密に評価し、未定義コマンドであった場合は、システム状態を破壊するような不正なメモリアクセスを直ちにブロックする。応答データとして 0x00 (またはデバッグ用に定義されたエラーコード 0xEE) を送信バッファにセットし、以後のPhase 2に対するダミークロックに対してはその無害な値を出力し続ける。内部のゲームパッド状態等の変数は一切更新しない。

#### **4.3.2 マスター側のタイムアウト検知 (デッドロック回避)**

* **異常条件:** Masterが CS を Low にしたまま、何らかの理由（マスター側のシステムフリーズ等）で長期間 SCK を送信してこない。または、Phase 1完了後にPhase 2を開始しないままスタックする。  
* **回復シーケンス:** スレーブであるCH32X035内部でハードウェアタイマー (TIMモジュール) を稼働させ、EXTIにより CS が Low になった瞬間にカウントを開始する。100ms以内に CS が High に戻らなかった場合、マスター側の異常とみなし、タイムアウトエラーとしてSPIモジュールをソフトウェアリセット (SPI\_CR1 レジスタの操作によるDisable/Enable等) し、通信状態を強制的に待機状態（Idle）に戻す 。これにより、スレーブ側が永遠に通信待ち状態でロックされることを防ぐ。

### **4.4 ティアリング (Tearing) 防止とロックフリー・ダブルバッファリング**

USB割り込みコンテキスト（ゲームパッドからの新規データ受信と解析）と、SPI割り込みコンテキスト（メインMCUからのデータ読み出し要求）は完全に非同期で発生する。

* **異常条件 (データ競合):** USB処理ルーチンがグローバルなゲームパッド状態バッファ（3バイト）を書き換えている最中に、タイミング悪くSPI割り込みが発生して読み出しが実行された場合、前半のバイトは「新しいデータ」、後半のバイトは「古いデータ」が混ざった不整合なデータ（ティアリング現象）がマスターに送信されてしまう。  
* **解決策 (正常系への昇華):** この問題を防ぐために割り込み禁止命令 (DI/EI) を多用すると、USB通信におけるミリ秒単位のクリティカルなタイミング要件を満たせなくなり、USBの切断を誘発する。したがって、割り込みを一切停止させない **「ロックフリー・ダブルバッファ（ピンポンバッファ）」** を実装してデータの整合性を100%確保する。

**ダブルバッファ実装シーケンス:**

1. メモリ上に同一サイズのバッファAとバッファBを用意し、ポインタ変数 read\_ptr と write\_ptr を管理する。  
2. USB処理ルーチンは、常に write\_ptr が指す裏バッファに対して最新の解析済レポートを書き込む。この最中、SPI通信からアクセスされることはない。  
3. USBバッファの書き込みが完全に完了した後、アトミックな操作（単一のポインタスワップ命令）によって write\_ptr と read\_ptr を入れ替える。  
4. SPI割り込みルーチン（およびDMA）は、コマンド受信時、常に最新の read\_ptr が指す表バッファの先頭アドレスを送信元アドレスとしてセットし、データを読み出す。  
   これにより、いかなる過酷なタイミングで非同期割り込みが衝突しても、データが途中で混ざることは物理的に不可能となる。

**5\. 統合的なシステム異常とフェールセーフ (System-level Failsafes)**

通信プロトコル以外の、物理的・環境的要因によるシステム全体の異常系定義と保護メカニズム。

### **5.1 VBUS過電流・短絡 (Overcurrent / Short Circuit) 保護**

* **異常条件:** ユーザーが故障したUSBデバイス、ショートしたケーブル、あるいは規格外の大電流を引き込むデバイスを接続した結果、システムの5V電源ラインやCH32X035のVDD (3.3V) 生成元に深刻な過負荷が生じ、電圧降下 (Brown-out) や発熱を引き起こす。  
* **保護要件とソフトウェア対応:** 本質的な過電流保護は、PA0 (Load Switch EN) によって制御される外部のロードスイッチIC (例: TPS2052, MIC20xx 等の電流制限機能付きUSB電源スイッチ) のハードウェア機能に依存する。 基板設計において、ロードスイッチの異常検知ピン (FAULT / OCP: Over-Current Protection, アクティブLow) をCH32X035の空きGPIO（例: PB12等）に接続し、外部割り込み (EXTI) で監視することを強く推奨する 。 過電流割り込みを検知した場合、ファームウェアは**即座に PA0 \= Low** を出力してVBUSの給電を強制遮断し、USBスタックを停止して再試行を禁止する。この「過電流ロック状態」はシステムリセットがかかるまでソフトウェア的にラッチされ、PA1とPA2のLEDを同時かつ高速に点滅させることで、ユーザーに「危険なデバイスが接続された」ことを警告する。

### **5.2 ブラウンアウト (BOD) / PVDリセット**

* **異常条件:** メインシステムの電源投入・切断時の過渡応答、またはUSBデバイス接続時の突入電流による瞬間的な電圧降下により、MCUの動作電圧範囲を下回る。  
* **回復シーケンス:** CH32X035に内蔵されている PVD (Programmable Voltage Detector) または POR/PDR (Power On/Down Reset) 回路を有効化する 。Vdd電圧が約 2.8V 未満に降下した場合、MCUは不安定な状態でのプログラム実行を防ぐため、ハードウェアレベルで安全にシステムリセットをかける。リセット後は周辺回路（SPI/USB）のステートマシンが初期化されるため、不定な信号を出力してメインMCUを混乱させることを防止できる。

### **5.3 EMI・サージノイズによるシステムフリーズとウォッチドッグ (WDT)**

* **異常条件:** 外部からの強烈な電磁干渉 (EMI) や静電気放電 (ESD) により、CPUのプログラムカウンタが書き換わり予期せぬメモリアドレスへジャンプする、あるいは割り込みフラグのクリア失敗により無限ループに陥る 。

* **回復シーケンス:** システムの完全停止を防ぐため、ハードウェア機能である **IWDG (Independent Watchdog Timer)** を必ず有効化して運用する。ファームウェアのメインループ内の最終段に、IWDGのカウンタをリロード (Feed / Kick) する処理を配置する。タイムアウト周期は約 **500ms** に設定する。  
  USBのSTALL復帰処理のデッドロック、SPIのCS割り込みのフリーズ、またはノイズによる暴走によってメインループが500ms以上停止した場合、IWDGがオーバーフローし、ハードウェアが自動的にチップ全体を再起動 (System Reset) する。再起動後、初期化プロセスを経て通常の待機状態へ自動復旧する。

**6\. コマンド体系とSPI入力データフォーマット**

メインMCUとやり取りするデータ構造の定義。通信は上記の「フェーズ分離」と「CSリカバリ」によって保護されていることを前提とする。

### **6.1 SPI コマンド体系**

| ID (MOSI) | 名称 | 返答サイズ | 詳細・異常系時の挙動 |
| :---- | :---- | :---- | :---- |
| **0x01** | **GET\_PAD\_INPUT** | 3 bytes | 解析済みの3バイト入力データ。USB通信異常時は、最後に取得した正常値を維持し続ける。ただし完全な「切断判定」を下した場合は、全ビットを初期値（ボタン離上、スティック中立）にクリアして送信する。 |
| **0x02** | **GET\_PAD\_RAW** | 可変 | USB生レポートデータをそのまま転送する。データ長は必ず事前に 0x03 で確認すること。 |
| **0x03** | **GET\_PAD\_RAW\_LEN** | 1 byte | 現在接続中のパッドの生レポート長を返す。未接続時や列挙失敗時は 0x00 を返す。 |
| **0x10** | **GET\_SYS\_STATUS** | 1 byte | システムの内部ステートマシン状態を返す。 0x00: 未接続 (Idle) 0x01: 接続済・正常通信中 0x02: 列挙中 (Enumerating) 0xEE: USB致命的異常 (列挙失敗、バブルエラー継続等) 0xEF: VBUS異常 (過電流検知等) |
| **0x11** | **GET\_DEVICE\_ID** | 4 bytes | 接続デバイスの VID (2 bytes) \+ PID (2 bytes) を返す。未接続時は 0x00000000。 |
| **0xFE** | **GET\_FW\_VERSION** | 2 bytes | ファームウェアバージョン (Major \+ Minor) を返す。(例: v2.0 なら 0x02, 0x00) |

### **6.2 SPI入力データフォーマット (3バイト固定)**

0x01 コマンドに対して送出される、CH32側で標準化・マッピング解析済みの24bitデータ。一般的なDirectInput/XInputゲームパッドの主要要素を網羅する。

| Byte | Bit 7 | Bit 6 | Bit 5 | Bit 4 | Bit 3 | Bit 2 | Bit 1 | Bit 0 |
| :---- | :---- | :---- | :---- | :---- | :---- | :---- | :---- | :---- |
| **0** | A (Cross) | B (Circle) | X (Square) | Y (Triangle) | L1 (Bumper) | R1 (Bumper) | L2 (Trigger) | R2 (Trigger) |
| **1** | Start / \+ | Select / \- | L3 (Stick) | R3 (Stick) | D-Pad 上 | D-Pad 右 | D-Pad 下 | D-Pad 左 |
| **2** | L-Stick 上 | L-Stick 右 | L-Stick 下 | L-Stick 左 | R-Stick 上 | R-Stick 右 | R-Stick 下 | R-Stick 左 |

*備考:* アナログスティックの値は、設定された閾値（デッドゾーン）を超えた場合にデジタルな「方向」としてビットが立つようにファームウェア内で2値化処理される。生のアナログ値が必要な場合は 0x02 コマンドを用いてRAWレポートを取得し、メインMCU側で独自にパースすること。

**7\. ステータスLEDインジケーター定義**

システムの現在の状態、および発生している異常の種類を開発者やエンドユーザーが物理的に視覚把握できるよう、PA1 (PadEnable) および PA2 (Status) に接続されたLEDの点灯パターンを以下のように厳密に定義する。

| システム状態 | PA1 (PadEnable) | PA2 (Status) | 詳細説明・想定される異常内容 |
| :---- | :---- | :---- | :---- |
| **電源投入・初期化中** | 両方点灯 | 両方点灯 | 起動直後のハードウェアセルフテスト中。(約200ms) |
| **正常: 未接続** | 消灯 | 消灯 | VBUS=OFF。CCピンの電圧が vOPEN を示している。 |
| **正常: 接続済・待機** | **常時点灯** | 消灯 | デバイスの列挙に成功し、ポーリング通信が確立されているが、ユーザー入力がない状態。 |
| **正常: 入力検知** | **常時点灯** | **点灯・明滅** | 正常通信中であり、かつゲームパッドから何らかの入力（ボタン押下、スティック操作）が行われている状態。入力の変化に合わせて明滅する。 |
| **異常: 一時的通信エラー** | 1秒に1回点滅 | 消灯 | USBバブリングによるHalt、トグルエラー、またはSTALLが発生し、現在リカバリシーケンス（再試行やClear Feature）を実行中。回復すれば正常状態に戻る。 |
| **異常: 列挙失敗** | **高速点滅** (200ms) | 消灯 | USB列挙に3回失敗した、あるいは bInterfaceClass \= 0xFF などの非対応・未知のベンダー独自デバイスが接続され、フォールバックにも失敗した致命的エラー。物理的な再接続が必要。 |
| **異常: SPI同期エラー** | 状態を維持 | **高速点滅** | CH32X035側はUSBを正常に処理しているが、SPI通信においてCSとSCKの同期エラー（ビットシフト、ショートパケット、タイムアウト）を連続して検知した。マスター（RP2350等）側のSPI設定やソフトウェア遅延、配線ノイズに問題がある可能性が高い。 |
| **異常: ハードウェア障害** | 交互に高速点滅 | 交互に高速点滅 | VBUSの過電流 (OCP) 割り込みを検知した、または強烈なノイズによりウォッチドッグタイマー (IWDG) が作動しシステムが再起動された直後のラッチ状態。安全のためVBUSは遮断される。 |

**8\. 開発・ビルド環境仕様**

安定したビルドと再現性を確保するため、開発環境は以下の要件を満たすこと。

* **Target MCU:** WCH CH32X035G8U6 (QFN28 パッケージ)  
* **Architecture:** QingKe 32-bit RISC-V4C Core (RV32IMAC 拡張)  
* **Toolchain:** xPack RISC-V Embedded GCC 15.2.0-1 以降  
* **Build Tool:** GNU Make (Linux / Windows MSYS2 / macOS 互換)  
* **Firmware SDK:** WCH ch32x035\_evt (v1.0.0 以降を推奨、レジスタ定義およびUSBFS割り込みベクタの利用のため)  
* **Programmer/Debugger:** WCH-LinkE (PC18/DIO, PC19/DCK を介した 2-wire SDI 通信)

本仕様書に基づくファームウェアの実装においては、第3章に規定した「USBホストの異常系ハンドリング（タイムアウトとSTALL処理）」および、第4章に規定した「SPIのCS (NSS) エッジ駆動同期によるビットシフト対策」の項目をアーキテクチャの根幹として最優先でコーディングすること。これらの要件を厳密に満たすコードを記述することにより、メインMCUに極限の負荷がかかる環境下や、劣悪なノイズ環境下においても、一切のパケット崩壊や通信のデッドロックを起こさない、極めて強靭なブリッジデバイスが実現される。
